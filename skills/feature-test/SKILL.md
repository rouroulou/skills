# Feature Test — 功能测试 + 体验审计一体化流程

对指定功能模块执行一次完整的测试闭环：**文档驱动生成用例 → 真实环境执行 → 双审计 → 结构化报告**。
全程使用中文。如果项目有 CLAUDE.md / AGENTS.md，先通读并遵守其中的验收、Mock、路由守卫等规范。

## 执行契约（先读这里）

**本 skill 是纯评估性质**：像一位 QA + 体验设计师在验收别人的功能，角色是"发现并报告问题"，不是"实现或修复"。它的价值在于完整闭环，不是单纯跑 QA。以下都算执行失败：

- ❌ 只跑了功能用例就出结果，跳过了双审计 —— QA 用例只能发现"功能通不通"，发现不了"好不好用"。没有审计的报告对用户没有交付价值。
- ❌ 产出物是 CSV / JSON / 表格文件 / 对话内清单 —— 唯一合法的交付物是**一份 Markdown 测试报告**（格式见阶段 4），其中必须包含按优先级划分的问题和优化方案。任何其他格式都不是本 skill 的产出。
- ❌ 写测试代码（vitest / jest / playwright 的 `*.test.*` / `*.spec.*` 文件）—— "测试"在本 skill 中指**人工驱动浏览器逐条执行用例**，不是沉淀自动化测试套件。用户要单测会另行提出，不要自作主张。
- ❌ 修改业务代码、修 bug、commit、push、建 PR —— 发现问题只写进报告，由用户决定何时修。整个流程中唯一允许的文件改动：① 报告文件本身 ② 为覆盖极端场景临时扩展的 mock 数据（需在报告中说明，且不 commit）。

开始时用 TodoList 工具把四个阶段建成待办项，每完成一个勾掉一个；结束前对照文末「交付自检」逐项核对，有一项不满足就回去补完，不要提前结束。多轮对话中用户追问或补测时，以上契约仍然有效——不要因为"上一轮已经人工测过了"就转去写自动化测试或改代码。

## 阶段 0：环境探测

开始前先摸清项目：

1. **功能文档位置**：依次查找 `docs/`、`doc/`、`documents/`、仓库根目录下的 `*.md`（PRD、方案、需求文档）
2. **dev 服务器**：从 `package.json` 的 `dev` 脚本和 vite/next/webpack 配置中确定端口；用 preview_start 启动或确认已运行
3. **Mock 体系**：识别项目用的 mock 方案（MSW / json-server / 内置 mock 等），极端场景造数时遵循同一套机制
4. **路由守卫**：识别登录守卫 / 游客守卫路径，测试时用正确的登录态访问，避免误报
5. **审计 skill 可用性**：确认 available skills 列表中是否存在 `frontend-design-audit` 与 `ux-audit`。命名前缀因安装方式而异，都算可用：项目内置版是裸名（`ux-audit`）、随本插件分发的是 `feature-test:ux-audit`、独立插件版是 `dev-tools:ux-audit` / `frontend-design-audit:evaluate`——用列表里实际出现的名字调用。现在就确认，不要等到阶段 3 才发现没有

## 输入解析

用户会指定一个功能模块（如"测试 referral"、"测一下团队管理"）。按以下顺序定位功能文档：

1. 在文档目录下找与模块名匹配的文档
2. 若未指定模块或匹配不到，列出所有文档让用户选择
3. 同时参考项目的 API 文档中该模块相关的接口定义（如有）

如果用户给的是当前分支正在开发的功能且没有点名文档，用 `git log --oneline -5` 和分支名推断模块，再去文档目录找对应文档。

## 阶段 1：通读功能文档，生成测试用例

完整读取功能文档（不要只读开头），提取：

- **核心用户流程**：每条端到端路径（如：进入落地页 → 复制邀请链接 → 被邀人注册 → 奖励入账）
- **业务规则与边界值**：文档中的阈值、上限、状态机、风控规则、预算约束等
- **角色与权限**：不同角色（普通用户 / 管理员 / 未登录）各自能看到和操作什么

据此生成两类测试用例，直接在对话中列出，作为后续执行的 checklist（注意：这只是过程清单，不是交付物，不要把它导出成文件交差）：

### A 类 — 功能连通性用例
覆盖每条核心流程的 happy path：页面可达（HTTP 200 + DOM 渲染）、数据正确加载、表单提交成功、状态流转正确、跳转链路完整。

### B 类 — 极端场景用例
针对每个交互点至少考虑：

| 维度 | 示例 |
| --- | --- |
| 空态 | 列表无数据、用户无记录、首次进入 |
| 极值 | 超长文本/名称、金额为 0 / 极大值、分页最后一页 |
| 非法输入 | 特殊字符、XSS payload、负数、格式错误 |
| 状态异常 | 接口慢响应、接口报错、401/403 |
| 权限 | 未登录访问守卫路由、角色越权操作 |
| 国际化 | 多语言切换后文案完整、无 key 裸露、布局不溢出（项目有 i18n 时） |
| 并发/重复 | 重复点击提交按钮、快速切换筛选条件（验证不闪烁/不重复提交） |
| 响应式 | 窄屏（375px）下布局不塌陷 |

每条用例格式：`[A/B-序号] 标题 | 前置条件 | 操作步骤 | 预期结果`。

## 阶段 2：实际执行验证

进入本阶段时，先把报告骨架文件建好：在 `docs/test-reports/<模块名>-<YYYY-MM-DD>.md` 写入阶段 4 模板的五个章节标题和第 2 节的用例清单表格（结果列留空）。之后边测边回填——这样无论流程在哪里被打断，报告文件都已存在且有内容，不会出现"测完了却没有报告"的局面。

**禁止纸上谈兵**——每条用例必须在真实运行的应用上执行（用 preview/浏览器工具人工驱动，不是写测试脚本），不能仅靠读代码下结论。

1. 确认 dev 服务器运行（阶段 0 探测到的端口）
2. 每个待测路由先自验：HTTP 200、关键 DOM 节点已渲染、console 无 error、路由守卫状态正确
3. 按用例清单逐条执行：preview_snapshot 看结构、preview_click / preview_fill 做交互、preview_console_logs + preview_network 抓错误、preview_resize 验证响应式
4. Mock 数据不足以覆盖某极端场景时，按项目现有 mock 机制临时扩展造数，测完说明改动
5. 每条用例记录结果：✅ 通过 / ❌ 失败（附复现步骤、console/network 证据、疑似代码位置 `file:line`）

执行完不等于结束——此时你只完成了一半。接下来必须进入阶段 3，**不要在这里直接跳到写报告**。

## 阶段 3：双审计（不可跳过）

这是与普通 QA 流程的本质区别：功能用例验证"能不能用"，审计验证"好不好用"。两者都做完，报告才完整。

依次用 **Skill 工具**真实调用以下两个审计（必须看到 Skill 工具的调用和返回，禁止自己"模拟审计"或凭印象写审计结论）：

1. **`frontend-design-audit`**（插件版叫 `frontend-design-audit:evaluate`）— 对该模块的页面代码做可用性审计，只评估不改代码
2. **`ux-audit`**（插件版叫 `dev-tools:ux-audit`）— 以真实用户身份走查该模块的所有页面与交互

调用时把模块范围、涉及的路由路径、阶段 2 已发现的问题清单作为上下文传入，避免重复劳动。收集两份审计的发现，合并进最终报告（去重：同一问题多处发现只记一条，标注来源）。

仅当阶段 0 已确认环境中确实没有这两个 skill 时，才允许跳过对应审计——且必须在报告第 4、5 节明确写出"环境缺失 XX skill，本次未执行该审计"，并自己按该审计的视角（可用性启发式 / 真实用户走查）做一轮人工替代检查，把发现写进报告。静默跳过 = 执行失败。

## 阶段 4：生成测试报告（唯一交付物）

**本 skill 的唯一交付物是一份 Markdown 报告文件**，保存到 `docs/test-reports/<模块名>-<YYYY-MM-DD>.md`（目录不存在则创建；项目文档目录不叫 docs 时跟随项目惯例）。不要生成 CSV、JSON、Excel 或任何"测试批次/用例导出"类文件——用户要的是一份能直接拿去排期修复的报告，不是原始数据。

**必须使用以下结构，五个章节一个不能少**：

```markdown
# <模块名> 测试报告

> 日期 | 分支 | 测试范围（路由列表）| 功能文档版本

## 1. 执行摘要
- 用例总数 / 通过 / 失败，审计发现数
- 一句话结论（可发布 / 需修复后发布 / 阻塞）

## 2. 测试用例执行结果
（A 类、B 类两张表：用例 ID | 标题 | 结果 | 备注）

## 3. 待修复问题（按优先级）

### P0 — 阻塞（核心流程不通 / 数据错误 / 安全问题）
### P1 — 严重（主要功能受损但有绕行方案 / 明显体验断裂）
### P2 — 一般（边缘场景缺陷 / 不一致 / 视觉问题）
### P3 — 优化建议（锦上添花）

每个问题必须包含四段：
- **现象**：复现步骤 + 实际 vs 预期（附证据：console 报错 / 截图路径 / 网络请求）
- **原因分析**：定位到具体代码（`file:line`），说明根因而不是表象
- **行业通用做法**：同类产品（如 Stripe、Vercel、Linear 等成熟 SaaS/开发者平台）
  对这个场景的标准处理方式，说明为什么这是惯例
- **优化方案建议**：结合本项目技术栈与现有可复用组件，给出最小改动的具体修复方案

## 4. 审计结论汇总
- frontend-design-audit 核心发现（链接到第 3 节对应问题）
- ux-audit verdict 与核心发现（链接到第 3 节对应问题）
（若某审计因环境缺失未执行，在此注明并给出人工替代检查的结论）

## 5. 风险与未覆盖项
（本次没测到的场景及原因，如依赖真实后端、需要多账号等）
```

**优先级判定原则**：影响核心业务流程（注册、付费、资金相关）的问题一律 P0/P1；纯视觉、文案、动效问题不高于 P2；极端场景缺陷按"真实用户撞上的概率 × 后果严重度"评估。

**「行业通用做法」要求**：不要泛泛说"业界通常会做好错误处理"，要具体——例如"Stripe Dashboard 在表格空态会给出主操作 CTA 而非纯文字提示"、"Vercel 对危险操作要求输入项目名确认"。如果对某场景的行业惯例没有把握，用 WebSearch 查证后再写。

## 交付自检（结束前逐项核对）

全部满足才能结束本次任务，否则回到对应阶段补完：

- [ ] 所有生成的用例都已在真实运行的应用上执行（或在报告第 5 节说明为何无法执行），没有"应该没问题"式的跳过
- [ ] `frontend-design-audit` 与 `ux-audit` 都已通过 Skill 工具真实调用并拿到返回结果（或环境确实缺失，已在报告中注明并做了人工替代检查）
- [ ] 交付物是**一份 .md 报告文件**，已落盘到 `docs/test-reports/`；没有生成 CSV/JSON/Excel 等替代产物
- [ ] 报告包含全部五个章节，第 3 节每个问题都有完整四段（现象/原因/行业做法/优化方案）
- [ ] 没有新建任何 `*.test.*` / `*.spec.*` 文件，没有修改业务代码，没有 commit / push / 建 PR
- [ ] 最后回复中给出了报告文件路径 + P0/P1 问题的一句话摘要
