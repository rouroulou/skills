# PRD Restructure — 需求结论沉淀为标准六段式 PRD

把一轮需求梳理的结论（对话讨论、草稿文档、零散笔记）重构为团队可直接评审/排期的标准 PRD。
全程使用中文。如果项目有 CLAUDE.md / AGENTS.md，先通读并遵守其中的验收链接、路由守卫等规范。

## 执行契约（先读这里）

**本 skill 是文档整理性质**：把已经讨论清楚的需求"结构化沉淀"，不是替用户做产品决策。以下都算执行失败：

- ❌ 自行发明需求 —— 文档里的每一条功能、优先级、规划都必须能追溯到讨论结论或原始草稿；信息缺口列进「待确认问题」，不要编造补全。
- ❌ 截图缺失或用占位符 —— 第 1 节的页面截图必须是真实运行页面的截图（流程见阶段 2）；页面尚未开发的功能在截图位注明「本期新建页面，暂无截图」，不要贴无关图或空链接。
- ❌ 测试链接未自验就写进文档 —— 每个链接发布前必须实测可用（HTTP 200 + 关键 DOM 已渲染 + 路由守卫状态正确）。坏链接比没有链接更伤评审信任。
- ❌ 修改业务代码、commit、push —— 唯一允许的文件产出：① PRD 文档本身 ② 截图文件。
- ❌ 交付物散落 —— 唯一交付物是**一份 Markdown PRD**（六段式，见阶段 3），截图作为其引用的附件存放。

开始时用 TodoList 把三个阶段建成待办；结束前对照文末「交付自检」逐项核对。

## 阶段 0：收集输入

PRD 的内容来源按优先级：

1. **当前对话中的需求讨论**——这是最常见的入口（用户说"梳理完了，整理成 PRD"），完整回顾本次会话的讨论结论，包括被否决的方案（写进远期规划或不写）
2. **已有草稿文档**——用户指定的或文档目录（`docs/` 等）里对应模块的方案/PRD 草稿，作为重构底稿
3. **代码现状**——涉及已有功能改造时，查路由、页面、API 类型定义，确认"现状"描述准确

如果范围、优先级或角色划分在讨论中没有明确结论，先列出你的理解让用户确认关键缺口，再动笔——返工一份完整 PRD 比先问三个问题贵得多。

## 阶段 1：信息整形

动笔前先把素材按六段结构归位，明确：

- **范围切割**：哪些是本期必做（P0）、本期可做（P1/P2）、明确不做（→ 第 6 节）
- **角色 × 状态矩阵**：涉及哪些角色（如 未登录访客 / 普通用户 / 管理员），每个角色经历哪些状态场景（如 未注册→已注册→已付费），每个交叉点对应哪个页面/路由
- **页面清单**：本期涉及的所有前端页面/路由，标注「已有页面改造」还是「本期新建」——这份清单同时驱动截图和测试链接
- **数据实体**：涉及的核心数据对象、字段、状态机、接口（从 API 类型定义 / mock 数据 / 后端 DTO 中核实，不要凭记忆写）

## 阶段 2：截图与链接自验

第 1 节的截图和第 2 节的测试链接都依赖真实运行的应用：

1. 确认 dev 服务器运行（从 `package.json` 确定端口；用 preview_start 启动）
2. **截图**：对页面清单中每个「已有页面」逐个访问并 preview_screenshot，保存到 `docs/screenshots/<模块名>/<页面名>.png`，在 PRD 中以相对路径嵌入（`![](../screenshots/...)`）。截图前确认页面处于有代表性的状态（有数据、正确登录态），不要截骨架屏或报错页
3. **测试链接自验**：每个写进第 2 节的链接，逐一验证 HTTP 200、关键 DOM 节点已渲染、console 无 error、路由守卫与目标角色匹配（守卫路由用对应登录态访问，游客路由确保未登录态或带预览参数）。验不过的当场换路径或在文档中注明「依赖 XX 后可验」
4. 「本期新建」页面：截图位写「本期新建，暂无截图」，链接位写规划路由并标注「待开发」

## 阶段 3：生成 PRD（唯一交付物）

保存到文档目录（如 `docs/<模块名>-PRD-<YYYY-MM-DD>.md`，跟随项目文档命名惯例）。**必须使用以下六段结构**：

```markdown
# <模块名> PRD

> 日期 | 版本 | 作者/来源（本次需求梳理会话）| 关联文档

## 1. 本期需求范围及功能优先级

| 优先级 | 功能 | 说明 | 涉及页面 |
| --- | --- | --- | --- |
| P0 | ... | ... | ... |

（每个涉及页面附功能截图；新建页面注明「本期新建，暂无截图」）

## 2. 角色 & 状态场景 Roadmap

按角色分小节（如：未登录访客 / 普通用户 / 管理员）。每个角色：
- 状态场景流转（如：点击邀请链接 → 注册 → 首充 → 奖励入账）
- 各场景对应的页面/路由 + **已自验通过的测试链接**
- 本期里程碑划分（哪些场景第一批交付、哪些随后）

## 3. 功能明细

按功能模块逐个展开：交互规则、边界条件、文案、异常处理、埋点（如有）。
每条功能可直接作为开发任务拆分依据——写到"开发看了不用再来问"的颗粒度。

## 4. 数据结构

核心数据实体、字段表（字段名 | 类型 | 说明 | 约束）、状态机、关键接口的请求/响应结构。
与代码中已有类型定义对齐（注明来源文件），新增实体标注「本期新增」。

## 5. 待确认问题及风险（没有可省略本节）

| # | 问题/风险 | 影响 | 建议方案 | 待谁确认 |

## 6. 远期规划（没有可省略本节）

讨论中提及但本期不做的方向。每条必须明确标注「**本期暂不涉及**」，
避免评审时被误认为本期范围。
```

**写作要求**：

- 优先级、范围划分忠实于讨论结论；你的补充建议（如风险提示）放第 5 节并标注「整理者建议」
- 数据结构与代码事实对齐——字段名、类型从实际类型定义文件核实并注明出处
- 第 6 节的每一条都写清"为什么本期不做"（依赖未就绪/优先级不够/方案未定），这是评审最常追问的点

**交付动作**：

- 运行环境有输出/下载通道（Cowork outputs、云端会话）→ 把 PRD 复制进去，且**只放这一份文件**（截图通过相对路径随仓库走，不单独塞进下载通道）
- 本地 Claude Code → 落盘后给出可点击路径
- 最终回复以"文档位置 + 本期范围一句话摘要 + 待确认问题数"收尾

## 交付自检（结束前逐项核对）

- [ ] 六段结构完整（5、6 节无内容可省略，省略时在文档开头的关联说明中注明"无待确认问题/无远期规划"）
- [ ] 第 1 节每个已有页面都有真实截图，截图文件已落盘且相对路径可解析
- [ ] 第 2 节每个测试链接都实测通过（HTTP 200 + DOM + 守卫态），未通过的已注明原因
- [ ] 第 4 节数据结构与代码类型定义核对过并注明来源
- [ ] 第 6 节每条都标注了「本期暂不涉及」
- [ ] 文档中没有编造的需求——每条内容可追溯到讨论结论、草稿或代码事实
- [ ] 除 PRD 和截图外没有产生其他文件，没有 commit / push
- [ ] 最后回复给出了文档路径 + 范围摘要 + 待确认问题数
